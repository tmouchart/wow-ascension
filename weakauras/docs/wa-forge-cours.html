01

Le problème qu'on résout

Le serveur privé Ascension (realm Conquest of Azeroth) a 21 classes custom — Felsworn, Barbarian, Runemaster… — avec des sorts, ressources et specs qui n'existent pas dans WoW retail. Les packs WeakAuras existants (Luxthos etc.) ne peuvent donc pas fonctionner pour elles.

Notre but : générer nous-mêmes des strings d'import WeakAuras (les fameuses !WA:2!... qu'on colle en jeu) pour chaque classe — barres de ressource, rangées de cooldowns, procs, glows — sans jamais passer par le jeu pour les créer.

L'insight fondateur du projet : le format !WA:2! a été complètement rétro-ingénieré. On sait le lire ET l'écrire depuis Node.js, sans aucune dépendance npm. Tout le reste du repo est construit sur cette capacité.

La vue d'ensemble : deux étages, chacun bidirectionnel. Tout le repo tient dans ce schéma.

02

Anatomie d'une string !WA:2!

Une string WeakAuras n'est pas du JSON ni du base64 standard. C'est une table Lua passée dans une chaîne de 4 transformations :

Table Luala structure de données de l'aura
→
LibSerializesérialisation binaire taguée (v1)
→
LibDeflateraw deflate — sans header zlib/gzip
→
EncodeForPrintbase64 custom 6 bits little-endian
→
!WA:2! + …le préfixe + la string finale

Pour décoder, on parcourt la chaîne en sens inverse. Tout est implémenté dans weakauras/lib/wa-codec.js — le cœur du projet, zéro dépendance.

Les pièges du format (déjà résolus, mais bons à connaître)

  • Le charset EncodeForPrint : a-z A-Z 0-9 ( ) — minuscules d'abord, pas les chiffres. Se tromper d'ordre donne une erreur deflate cryptique (« invalid block type »).
  • LibSerialize : le premier octet est la version (1), puis un flux d'objets tagués — les bits de poids faible de chaque tag disent le type (entier 7 bits, string/table embarquée, type étendu…). Entiers en big-endian.
  • Les clés numériques : en Lua, load.class_and_spec.multi = {[262]=true} a une clé entière. En JSON on ne peut pas — donc le codec les préfixe "$n$262" pour les ré-émettre en entiers à l'encodage. Sans ça, les conditions de chargement classe/spec cassent silencieusement.
  • Notre encodeur est non-optimisant (pas de déduplication de strings) : les strings ré-encodées sont ~30% plus grosses que l'original, mais WeakAuras les importe sans broncher.

Le codec en pratique

# CLI
cd weakauras/lib
node wa-codec.js decode mystring.txt    # !WA:2! -> mystring.decoded.json
node wa-codec.js encode edited.json     # JSON    -> edited.import.txt
node wa-codec.js roundtrip mystring.txt # decode -> encode -> decode, égalité profonde

// En module
const { decodeWA, encodeWA } = require('./lib/wa-codec.js');
const out = decodeWA(str);
// GOTCHA : l'enveloppe {d, c, m, s, v} est sous out.data, PAS à la racine
03

La structure d'une WeakAura décodée

Une fois décodée, une aura exportée est une enveloppe avec cinq champs :

{
  "d": { /* la région RACINE : le group conteneur */ },
  "c": [ /* TOUTES les régions, à plat (enfants inclus) */ ],
  "m": "d",        // marqueur
  "s": "5.20.2",   // version WeakAuras ciblée
  "v": 2000        // version du format d'export
}

Chaque entrée de c est une région. On utilise 4 types de régions :

Type de régionRôleTemplate
groupLe conteneur racine statique ; controlledChildren liste les ids de ses enfants dans l'ordre d'affichage.templates/group.json
dynamicgroupUne rangée/colonne qui arrange ses enfants automatiquement (les rangées de cooldowns, les colonnes latérales). Utilise notre Lua customGrow pour centrer + wrapper.templates/dyngroup.json
aurabarUne barre de progression (ressource, HP, uptime d'un buff, cases de points).templates/bar.json
iconUne icône de sort (cooldown, proc, indicateur de buff) avec son subglow.templates/icon.json

L'astuce fondatrice : cloner, pas construire

Une région WeakAuras 5.x a des dizaines de champs internes obligatoires. Plutôt que de les deviner, on clone des régions connues-bonnes extraites de packages Luxthos décodés (les templates ci-dessus), puis on override uniquement ce qui compte : id, position, couleurs, triggers, conditions. Impossible d'oublier un champ requis.

Dans une région : triggers, conditions, sub-regions

Trois mécanismes font tout le comportement dynamique :

  • Triggers — la source de données. Sur Ascension, la détection de buffs passe toujours par le trigger aura2 (par nom exact), car C_UnitAuras n'existe pas sur ce client. Les autres : Power (ressource), Health, Cooldown Progress (Spell), plus nos triggers Lua custom (targetExecuteTrigger, powerAtLeastTrigger) pour ce que le client ne filtre pas nativement.
  • Conditions — des règles « si variable du trigger X, alors changer la propriété Y ». Exemple : stacks >= 4 → peindre le remplissage de la case ; onCooldown == 1 → desaturate. Les checks se combinent avec un wrapper AND.
  • Sub-regions — des couches empilées sur la région : texte (subtext), et surtout subglow, le glow qu'une condition allume/éteint.

Extrait (simplifié) d'une condition réelle — « quand le buff manque, glow rouge » :

{
  "check": { "trigger": 1, "variable": "buffed", "op": "==", "value": "0" },
  "changes": [
    { "property": "sub.3.glow", "value": true },
    { "property": "sub.3.glowColor", "value": [1, 0.1, 0.1, 1] }
  ]
}
La taxonomie des glows — le style = l'urgence, la couleur = le sens. Action Button Glow (buttonOverlay) = « agis MAINTENANT » : blanc pour un proc/sort prêt, or/orange pour un dump de ressource. Pixel glow = état passif : couleur de classe pour « mon buff défensif est actif », rouge pulsant pour « un buff d'entretien est tombé ».
04

Le SPEC — notre DSL déclaratif

Écrire des régions à la main (même via des builders) reste verbeux et error-prone. Depuis le 14/07, chaque classe est décrite par un seul fichier classes/<nom>/spec.json : le SPEC. C'est un DSL déclaratif qui parle en éléments de gameplay (« une barre d'énergie », « une rangée de CDs », « un proc quand tel buff est up ») — plus jamais en champs de régions.

Point crucial : le SPEC est la source unique de vérité, consommée telle quelle par deux consommateurs : le build Node (node classes/<nom>/spec.js → dist/) et la web app (chargé via import.meta.glob, édité dans le store zustand, généré côté client). On ne duplique jamais les données de spec dans web/src.

Un SPEC réel, annoté — Felsworn Tyrant

{
  "slug": "felsworn",
  "id": "Felsworn Tyrant SPEC",          // racine des ids -> uids stables
  "global": { "barWidth": 250, "iconSize": 26, "gap": 3 },
  "stack": [                              // la pile centrale, de HAUT en BAS

    // Un proc : Fel Fireball ne s'affiche QUE si le buff "Carve" est up,
    // avec le glow blanc "agis maintenant"
    { "kind": "procRow", "icons": [
      { "label": "Fel Fireball", "spell": "Fel Fireball", "byName": true,
        "buff": "Carve", "glowType": "buttonOverlay" },
      // Un proc "execute" : visible quand la cible passe sous 35% HP
      { "label": "Tyrants Gaze", "spell": 805240, "execute": 35 }
    ] },

    // La rangée de cooldowns : grise sur CD, glow Pixel vert si le buff est actif
    { "kind": "cdRow", "icons": [
      { "label": "Hateforged Barrier", "spell": 705129,
        "glow": { "type": "buff", "buff": "Hateforged Barrier", "glowType": "Pixel" } },
      { "label": "Chaos Rush", "spell": "Chaos Rush", "byName": true, "charges": true }
    ] },

    // Barre d'uptime : vert -> jaune -> rouge, glow + warning si le buff tombe
    { "kind": "uptimeBar", "buff": "Inner Demon", "warnText": "Inner Demon" },

    // Ressource : powerType 3 = Energy, dégradé or
    { "kind": "powerBar", "powerType": 3, "hi": [1, 0.88, 0.15, 1] },

    // 6 cases Felfury ; glow or à 6 stacks SAUF si Inner Demon est deja up
    { "kind": "stacks", "auraNames": ["Felfury"], "count": 6,
      "capGlow": { "at": 6, "unlessBuff": "Inner Demon" } },

    { "kind": "healthBar" },
    { "kind": "cdRow", "secondary": true, "icons": [ /* défensifs, plus petits */ ] }
  ]
}

Version abrégée de classes/felsworn/spec.json. Remarque : zéro yOffset, zéro trigger, zéro condition — tout ça est dérivé.

Voilà ce que ce SPEC décrit visuellement (la « pile centrale », largeur 250px) :

procs · CDs · uptime Inner Demon · Energy · 6× Felfury · HP · CDs secondaires

Le vocabulaire d'éléments (la taxonomie)

KindCe que c'estCe qui est généré
powerBarRessource primaire (Energy, Mana, Rage…)aurabar + trigger Power, texte %p
healthBarHP du joueuraurabar rouge + trigger Health
stacksPoints/combo via stacks d'un buff (Felfury 0–6)N× aurabar cases ; condition stacks>=i peint la case i
chargeStacksCharges d'un sort (Runeblade 0–3)N× cases ; condition charges>=i
stackBarRessource en aura-stacks continue (Insanity 0–100)une barre pilotée par stacks
uptimeBarBuff d'entretien à maintenir (Inner Demon)barre countdown ; vert→jaune→rouge par expirationTime ; tombé = glow rouge + warn

cdRowRangée de cooldownsdynamicgroup d'icon ; désaturé sur CD ; 1 règle de glow max chacune

Réflexe à garder : avant d'éditer un spec ou d'ajouter un kind, lire docs/spec-dsl-reference.md — c'est le lexique complet (chaque champ de chaque kind, la table des clauses de proc, le mapping trigger→variables de condition, la math du layout).
05

Compilation : specToParts — du SPEC aux régions

Le compilateur vit dans lib/spec-builder.js. Il fait trois choses, dans l'ordre :

  1. Validation — kind inconnu, champ manquant, id de région dupliqué → erreur bruyante immédiate. Un SPEC invalide ne produit jamais une string silencieusement fausse.
  2. Layout engine — c'est lui qui calcule toute la géométrie : il descend la stack du haut vers le bas, additionne hauteurs + gap, et produit les yOffset de chaque région. Largeur unique 250px ; les cases de points dérivent leur largeur du nombre demandé ; les rangées d'icônes wrappent au-delà de maxWidth. Tu ne poses jamais une coordonnée à la main.
  3. Auto-wiring — chaque kind est traduit en appels aux builders de lib/builders.js (cooldownIcon, uptimeBar, segmentBar, makeDynGroup…), qui clonent les templates Luxthos et branchent triggers + conditions + subglows. C'est ici que « capGlow at 6 unlessBuff Inner Demon » devient un check AND stacks==6 && buffed(Inner Demon)==0 → sub.N.glow=true.

Puis l'assemblage : buildPackage

specToPartsSPEC → liste de régions + racine
→
assembleTopenveloppe {d, c, m, s, v}
→
encodeWAl'enveloppe → !WA:2!
→
round-tripdecode(encode(x)) == x sinon REFUS d'écrire
→
dist/<classe>-spec.import.txt (+ rotation .prev)
Les uids sont déterministes. Chaque région a un uid dérivé de son id stable via uidFor(). Résultat : ré-importer une nouvelle version en jeu propose « Update » au lieu de créer un doublon. C'est pour ça qu'on ne renomme jamais un id à la légère (et que le spec.json du cultist garde son id générique).
06

Décompilation : waToSpec — le chemin inverse

lib/wa-to-spec.js est l'inverse exact de specToParts : il prend les régions décodées d'une string !WA:2! et reconstruit le SPEC éditable. C'est ce qui alimente le bouton « Import string » de la web app (via POST /api/import côté backend) : tu colles une string générée par la Forge, tu récupères son SPEC dans l'éditeur.

Sa philosophie : fidèle, jamais devin. Il ne sait inverser que les formes exactes que nos builders émettent. Une région qu'il ne reconnaît pas (une WA tierce, ou un pattern qu'aucun package n'utilise encore) fait throw bruyamment plutôt que de produire un SPEC plausible mais faux. C'est un choix délibéré, aligné sur le principe « no guessing » du projet.

La boucle complète qui doit toujours tenir :

spec.json ──specToParts──> régions ──waToSpec──> SPEC' ──specToParts──> régions'

assert(régions == régions')   // byte-identique, pour les 5 classes

C'est exactement ce que vérifie node tools/wa-to-spec-roundtrip.mjs.

07

Les garde-fous — pourquoi on peut refactorer sans peur

Trois filets de sécurité indépendants verrouillent le pipeline :

Garde-fouCe qu'il vérifieQuand il tourne
self round-tripChaque string produite se re-décode à l'identique (decode(encode(x)) == x). Échec = buildPackage refuse d'écrire.À chaque build, automatiquement
verify-unchangedRebuild TOUS les packages (build.js + spec.js) et compare le JSON décodé aux snapshots golden (tools/golden/), octet par octet. Un refactor du moteur qui change une sortie = rouge.Après tout refactor de lib/ ; --snapshot pour re-baseliner un changement voulu
wa-to-spec-roundtripChaque spec.json survit au cycle compile → décompile → recompile en régions byte-identiques.Après tout changement du DSL, de spec-builder ou de wa-to-spec

À ces filets s'ajoutent deux règles de discipline : ne jamais éditer un .import.txt à la main (on édite le SPEC et on rebuild), et ASCII uniquement dans le texte généré (le codec ne round-trip pas l'UTF-8 multi-octets — un « ⚠ » dans un label fait échouer le build).

08

La web app & le déploiement

Live sur wa-forge.fly.dev. L'éditeur (Vite/React, weakauras/web/) est construit autour du même SPEC :

  • Les presets = littéralement les fichiers classes/*/spec.json, chargés par import.meta.glob. Aucune duplication.
  • L'état éditable vit dans un store zustand (src/store.ts) — jamais dans du state local de composant.
  • La génération de la string est 100% côté client : specToParts + assembleTop + le codec web (le moteur builders-core.js est isomorphe Node/navigateur).
  • Le backend Hono (server/server.mjs) ne sert que 2 routes : POST /api/import (string → SPEC, la décompilation) et POST /api/agent (édition du SPEC en langage naturel via LLM).
  • UI : shadcn/ui + Tailwind v4 uniquement, tokens sémantiques (jamais de couleur en dur), icônes lucide, classes fusionnées avec cn().
Déploiement : git push origin main = déploiement fly.io automatique (GitHub Action, build Docker distant). Piège connu : le CI ne voit que les fichiers commités — un fichier runtime laissé untracked marche en local et crash la prod (déjà vécu avec wa-to-spec.js). Autre règle : npm run dev tourne déjà en permanence chez toi — on ne le relance jamais, on lit dev.log pour débugger.

Gotchas Ascension — ce que le client custom impose

Jamais de « Spell Known » en load condition. IsSpellKnown sur un spellId custom CoA fait throw "Invalid spell slot", et cette erreur Lua avorte toute la boucle de chargement de WeakAuras : /wa ne s'ouvre plus du tout. Confirmé en jeu le 20/07. Le gate gateSpellKnown est OFF par défaut — ne pas le réactiver.
  • Détection de buffs = aura2 par nom exact, toujours. C_UnitAuras n'existe pas ; le Lua custom ne peut pas lire les auras. Corollaire : un nom de buff ≠ nom du sort échoue silencieusement — d'où la checklist AURA-NAMES-CHECKLIST.md (23 noms à cocher en jeu).
  • Les filtres natifs Health % et Power % ne gatent pas sur ce client — d'où nos triggers Lua custom targetExecuteTrigger (execute <35%) et powerAtLeastTrigger (montrer une icône à partir de N ressource).
  • Icône d'un sort en fallback : iconSource:-1 (auto) + un cooldownTrigger dont displayIcon est le chemin de texture. Jamais iconSource:0 avec un chemin — le mode manuel exige un fileID numérique, sinon → « ? ».
  • Deux angles morts avant de scaler aux 16 classes restantes : (1) les noms d'auras non validés en jeu ; (2) l'index de ressource (powerType) confirmé seulement pour barbarian et felsworn — les 19 autres sont des hypothèses (registry/resource-model.json).

En une phrase

Le repo est un compilateur bidirectionnel : un DSL de gameplay (spec.json) compilé en régions WeakAuras clonées de shapes connues-bonnes, encodées en !WA:2! par un codec maison — chaque étage inversible, chaque sortie verrouillée par un round-trip ou un golden snapshot, et une web app qui consomme exactement les mêmes specs.
